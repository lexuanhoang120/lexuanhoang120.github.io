Restaurant Traffic Forecasting

CADS - FPT Corporation · 2024.01 - 2024.05

A machine learning time-series forecasting pipeline for predicting next-month daily guest traffic at restaurant level for the Golden Gate restaurant chain. The project combines sales-data preprocessing, restaurant behavior clustering, calendar and holiday feature engineering, and separate XGBoost forecasting strategies for stable and high-volatility restaurant groups.

Demo

The demo shows how the forecasting pipeline generates daily guest-count predictions for the next month. Users can select a forecast month, run preprocessing, generate forecasts for different restaurant groups, and aggregate the final prediction output.

Demo Flow

  1. Select the target forecast month, for example 2024-05.
  2. Load restaurant sales, restaurant metadata, holiday, pre-holiday, lunar-calendar, and cluster data.
  3. Preprocess sales data by removing invalid values, handling missing traffic, and detecting abnormal points.
  4. Cluster restaurants by weekly seasonality and traffic volatility.
  5. Run the Global model for lower-complexity restaurant patterns.
  6. Run the Local model for higher-complexity restaurant patterns.
  7. Merge both outputs into one final restaurant-level daily forecast file.

Approach

Problem Definition

The objective is to forecast the number of customers visiting each restaurant on each day of the next month. Input data includes daily sales records such as guest count, total bills, total sales, and loyalty sales, together with restaurant metadata and calendar-related data.

Data Sources

  • sales_store: daily restaurant sales and guest-count history.
  • d_restaurant: restaurant profile, opening date, closing date, brand, and operating metadata.
  • d_date: date dimension, weekday, and holiday information.
  • moon_calendar: lunar-calendar dates used for Vietnamese calendar effects.
  • holiday_weight: estimated impact weight of holidays for each restaurant or brand.
  • pre_holiday: indicators for days before major holidays.
  • cluster_km: restaurant behavior clusters based on traffic seasonality and volatility.

Preprocessing

  • Removed rows with null, zero, or negative guest_count.
  • Detected unusually high or low traffic values using a Prophet-based abnormal-data handling flow.
  • Created a continuous restaurant-date panel so missing dates could be handled consistently.
  • Filled missing guest count using the same weekday from the previous week when available.
  • Used restaurant-year average guest count as a fallback for remaining missing values.
  • Filtered restaurants based on operating status, opening/closing dates, location, brand, and minimum data history.

Feature Engineering

  • Restaurant features: restaurant cluster label based on traffic complexity.
  • Calendar features: day of week, weekend flag, month, and year.
  • Holiday features: holiday indicator, pre-holiday indicator, and holiday impact weight.
  • Lunar-calendar features: beginning-of-month and mid-month lunar date indicators.
  • Lag features: historical guest count from previous days, including short-term and weekly-cycle lags.
  • Statistical features: mean, median, min, and max guest count from recent historical windows.

Restaurant Segmentation

Restaurants are clustered into 5 groups using weekly periodicity and traffic fluctuation. Weekly periodicity is calculated using a Fourier-based approach, while fluctuation is derived from historical traffic variation. These clusters separate restaurants with clear weekly patterns from restaurants with unstable or highly volatile traffic behavior.

Modeling Strategy

  • Global model: XGBoost model for restaurants with clearer weekly cycles and lower-complexity traffic patterns.
  • Local model: XGBoost model for restaurants with unclear weekly cycles and higher traffic volatility.
  • Training window: historical data from January 2022 to the month before the forecast month.
  • Test window: the target forecast month.
  • Validation period: July 2022 to December 2023 for selecting model parameters.

Evaluation Metrics

  • MAPE: Mean Absolute Percentage Error.
  • WAPE: Weighted Absolute Percentage Error, used as the main evaluation metric.

Key Results

  • Daily WAPE improved from 25.3% to 22.0%.
  • Baseline restaurant daily WAPE improved from 24.6% to 21.3%.
  • Non-baseline restaurant daily WAPE improved from 29.0% to 25.3%.
  • Weekly WAPE improved from 17.5% to 13.6%.
  • Monthly WAPE improved from 15.5% to 10.9%.
  • The new forecasting flow improved error by about 3-4 percentage points across daily, weekly, and monthly evaluation levels.

Feature Importance

The most influential features were mainly historical traffic lag features, traffic statistics, and holiday-related features. This indicates that restaurant traffic is strongly driven by recent historical patterns, weekly cycles, and holiday effects.

Output

The final output is a merged prediction file containing daily guest-count forecasts for restaurants in the target forecast month, generated by combining results from both the Global and Local model pipelines.
